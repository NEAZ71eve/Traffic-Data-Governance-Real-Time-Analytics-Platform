## 1. 项目背景与业务痛点

### 1.1 业务背景

随着智慧城市建设深入推进，城市交通管理部门面临海量实时数据的处理挑战：

- 全市部署 **5000+ 车辆卡口**、**2000+ 路况传感器**、**1000+ 智能信号设备**
- 日均产生 **500 万条**车辆通行记录、**300 万条**设备心跳数据
- 传统离线报表 T+1 产出，无法支撑**实时拥堵疏导**和**设备故障快速响应**

### 1.2 六大业务痛点

| 序号 | 痛点 | 现状 | 影响 |
|------|------|------|------|
| **P1** | 数据不可见 | 数据散落各业务系统，无统一视图 | 决策滞后 4-8 小时 |
| **P2** | 质量无保障 | 重复、空值、脏数据混入分析链路 | 指标失真，错误率 >5% |
| **P3** | 实时性不足 | 离线批处理 T+1，高峰期无法及时响应 | 拥堵处置延迟，市民投诉激增 |
| **P4** | 设备运维盲区 | 设备离线/异常靠人工巡检发现 | 故障平均发现时间 >2 小时 |
| **P5** | 数据孤岛 | 各部门自建系统，口径不一致 | 跨部门数据对不上，管理层质疑 |
| **P6** | 智能化缺失 | 无异常自动检测、无智能问答 | 全靠人工 SQL 提数，效率低下 |

### 1.3 项目目标

构建**一个平台**实现交通数据**全域采集 → 实时治理 → 智能分析 → 统一服务**的完整闭环，做到：

- **秒级感知**：端到端延迟 < 5 秒
- **精准分析**：数据质量识别率 > 98%
- **智能运维**：设备故障自动发现 / 拥堵自动预警
- **统一口径**：一套数仓、一份数据、一张看板

## 2. 解决方案概述

### 2.1 双链路架构

平台采用 **实时链路 + 离线链路** 双轨并行架构：

```
┌──────────────────────────────────────────────────────────────┐
│                        实时链路 (秒级)                         │
│  Kafka ──▶ Flink ──▶ Redis ──▶ Superset 实时看板             │
│              │                                                │
│              └──▶ Kafka Alert ──▶ 钉钉/邮件告警               │
├──────────────────────────────────────────────────────────────┤
│                        离线链路 (天级)                         │
│  Maxwell/DataX ──▶ HDFS ──▶ Hive ODS                          │
│                              │                                │
│      ┌───────────────────────┼───────────────────────┐        │
│      ▼                       ▼                       ▼        │
│   DIM (SCD2)              DWD (清洗)              AI模块       │
│      │                       │                                │
│      └───────────┬───────────┘                                │
│                  ▼                                            │
│              DWS (聚合)                                       │
│                  │                                            │
│                  ▼                                            │
│              ADS (指标) ──▶ Superset 分析看板                  │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 四层能力矩阵

| 能力层 | 覆盖范围 | 核心组件 |
|--------|---------|---------|
| **采集层** | 车辆卡口 / 路况传感器 / 设备心跳 / 告警日志 | DataX + Maxwell + Flume |
| **计算层** | 实时流计算 + 离线批处理 + 数据回溯 | Flink + Hive + DolphinScheduler |
| **治理层** | 质量监控 / 血缘追踪 / 权限管控 / 告警分发 | Griffin + Atlas + Ranger |
| **服务层** | 运营看板 / 运维看板 / 决策看板 / 质量看板 | Superset + Redis + DingTalk |

## 3. 业务场景与用例

### 3.1 场景一：高峰期拥堵实时预警

| 阶段 | 动作 | 耗时 |
|------|------|------|
| 数据产生 | 路况传感器每 30s 上报车速/车流 | — |
| 实时计算 | Flink 5min 窗口聚合 | < 3s |
| 拥堵判定 | 5 级阈值（车速<10km/h + 车流>2000/h） | < 1s |
| 告警推送 | 钉钉群推送拥堵路段+级别+建议 | < 1s |
| **端到端** | **数据产生 → 告警送达** | **< 5s** |

### 3.2 场景二：设备故障自动发现与工单联动

| 阶段 | 动作 |
|------|------|
| 心跳监测 | 设备每 1min 上报状态 (CPU/内存/温度/在线) |
| CEP 检测 | Flink CEP 30s 内连续 3 次离线 → 触发告警 |
| 健康评分 | DWS 层计算四维加权评分 (在线率40% + 资源30% + 异常20% + 恢复10%) |
| 分级告警 | P0(宕机) / P1(高CPU) / P2(温度异常) / P3(网络波动) |
| 工单创建 | 钉钉机器人自动创建运维工单，推送责任人 |

### 3.3 场景三：数据质量全链路监控

| 指标 | 规则 | 阈值 |
|------|------|------|
| 完整率 | 关键字段空值率 | < 1% |
| 唯一率 | 主键重复率 | < 0.1% |
| 合法性 | 值域范围校验（车速 0-200 km/h） | 违反率 < 0.1% |
| 时效性 | Kafka Lag 积压 | < 10000 |

### 3.4 场景四：AI 辅助数据分析

| 能力 | 输入 | 输出 |
|------|------|------|
| NL2SQL | "昨天早高峰最拥堵的 10 条路" | `SELECT road_name, avg_speed FROM ads_top_jam_roads WHERE ...` |
| 异常检测 | Isolation Forest 历史数据分布 | 异常设备/异常时段标注 |
| ETL 生成 | 自然语言描述 | 完整 SQL DDL + ETL 脚本 |

## 5. 技术选型说明

### 5.1 选型原则

- **成熟稳定**：优先选择 Apache 基金会孵化毕业项目
- **社区活跃**：GitHub Star > 5000，issue 响应及时
- **生产验证**：有互联网公司大规模使用案例
- **团队能力**：Java/Python 技术栈，降低学习成本

### 5.2 组件选型对比

| 能力域 | 候选方案 | 最终选型 | 选型理由 |
|--------|---------|---------|---------|
| 消息队列 | Kafka / Pulsar / RocketMQ | **Kafka** | 生态最成熟，Flink 原生支持，8分区高吞吐 |
| 实时计算 | Flink / Spark Streaming / Storm | **Flink** | 真正的流处理，Exactly-Once 语义，CEP 库 |
| 离线数仓 | Hive / Spark SQL / ClickHouse | **Hive** | 与 Hadoop 生态集成，ORC+Snappy 列存 |
| 任务调度 | DolphinScheduler / Airflow / Azkaban | **DolphinScheduler** | 分布式、可视化 DAG、补数重跑 |
| 数据治理 | Griffin / Great Expectations | **Griffin** | 与 Hive 原生集成，准确率/完整性度量 |
| 数据血缘 | Atlas / DataHub / Amundsen | **Atlas** | 与 Hive Hook 自动采集，GraphDB 查询 |
| 权限管控 | Ranger / Sentry | **Ranger** | 细粒度行列级权限，Audit 审计 |
| 可视化 | Superset / Metabase / Grafana | **Superset** | SQL IDE、丰富图表、角色级权限 |
| 实时缓存 | Redis / Aerospike / Ignite | **Redis** | 简单可靠，Flink Redis Connector |

### 5.3 版本兼容矩阵

| 组件 | 版本 | JDK | 备注 |
|------|------|-----|------|
| Apache Flink | 1.15+ | 8/11 | 需 Hadoop 2.8+ classpath |

## 6. 核心模块详解

### 6.1 实时流计算 (Flink)

**三作业协同**：

```
Kafka traffic_vehicle ──▶ TrafficVehicleCount ──▶ Redis (车流缓存)
Kafka traffic_status  ──▶ TrafficCongestionDetection ──▶ Kafka alert_topic
Kafka device_status   ──▶ DeviceStatusCEP ──▶ Kafka alert_topic
```

| 特性 | 配置 |
|------|------|
| Checkpoint | 5min 间隔，EXACTLY_ONCE 语义 |
| State Backend | HashMapStateBackend + RocksDB（大状态） |
| Watermark | 30s BoundedOutOfOrderness |
| 重启策略 | 固定延迟 3 次，每次 60s |
| Savepoint | 手动触发，用于升级/迁移 |

**CEP 规则引擎**（DeviceStatusCEP）：

| 规则ID | 模式 | 条件 | 告警级别 |
|--------|------|------|---------|
| RULE_OFFLINE | 连续 3 次 | status=OFFLINE, 30s 窗口 | P0 |
| RULE_CPU_HIGH | 连续 5 次 | cpu_usage>90%, 60s 窗口 | P1 |
| RULE_TEMP_HIGH | 连续 3 次 | temperature>80°C, 60s 窗口 | P2 |

## 10. 常见问题 FAQ

### Q1: 这个项目适合什么规模的场景？
A: 日均百万级到千万级数据量，适用于中大型城市交通管理部门或智慧城市项目。架构可按需横向扩展。

### Q2: 最小部署需要多少资源？
A: 开发/测试环境 1 台 16C/32G 服务器即可跑通全部组件。建议 SSD 硬盘以提高 Hive 查询性能。

### Q3: 实时计算延迟为什么能到 <5 秒？
A: Kafka → Flink 流处理 → Redis 写入，全程内存计算，无磁盘 I/O 阻塞。拥堵判定逻辑简单，窗口+阈值即可。

### Q4: 如果 Flink 作业崩溃了怎么办？
A: Flink 配置了 Checkpoint (5min) + Savepoint 机制，崩溃后自动从最近 Checkpoint 恢复，Exactly-Once 语义保证数据不丢不重。3 次重启失败后触发告警。

### Q5: 数据回溯怎么做？
A: DolphinScheduler 支持补数功能。在 `dolphinscheduler_config.json` 中定义了回溯策略，选择日期范围即可自动重跑 ODS→ADS 全链路。

### Q6: 数据质量发现异常后会自动修复吗？
A: 不会自动修复（避免误改）。会根据配置自动推送钉钉/邮件告警给数据开发，由人工判断后触发重跑或数据修复流程。

### Q7: SCD2 拉链表是怎么实现的？
A: dim_road_zip 和 dim_device_zip 采用 SCD2 策略：每条记录有 `start_date` 和 `end_date`，变更时旧记录 `end_date` 设为当前日期，新记录 `start_date` 设为当前日期，`end_date='9999-12-31'` 表示当前有效。

### Q8: AI 模块的 NL2SQL 支持哪些查询？
A: 支持单表查询、简单多表 JOIN、聚合查询（SUM/COUNT/AVG）、分组排序（TOP N）、时间范围过滤。复杂嵌套子查询需要人工校验。

### Q9: 如何添加新的数据源？
A: 三步走：
1. 在 `config/kafka_topics.json` 新增 Topic 配置
2. 编写对应 ODS 层 SQL 建表脚本
3. 在 DolphinScheduler 中添加新任务节点

### Q10: 看板可以嵌入到第三方系统吗？
A: Superset 支持 iframe 嵌入和 API 调用，可通过 URL 参数传递筛选条件，实现与第三方系统的集成。
