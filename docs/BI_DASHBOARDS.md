## 1. 看板总览

| # | 看板名称 | 目标用户 | 刷新频率 | 图表数 | 数据源 | 一句话用途 |
|---|---------|---------|---------|--------|--------|-----------|
| 1 | 城市交通总览大屏 | 领导 / 运营分析师 | 5分钟 | 7 | Hive ADS | 一眼看懂全市交通状况 |
| 2 | 实时路况监控面板 | 运营分析师 | **5秒** | 6 | Redis (Flink) | 秒级感知拥堵和异常 |
| 3 | 设备运维监控大屏 | 运维工程师 | 1分钟 | 8 | Hive ADS | 设备故障第一时间发现 |
| 4 | 数据质量监控面板 | 数据开发工程师 | 10分钟 | 5 | Python报告 | 数据有没有问题一目了然 |

### 用户能看到什么？（一句话版）

| 看板 | 能看到什么 |
|------|-----------|
| 总览大屏 | 全市堵不堵、今天多少车、哪里最堵、高峰几点 |
| 实时监控 | 数字每秒在跳、拥堵路段实时变红、流量突增立刻报警 |
| 设备运维 | 哪些设备在线/离线、CPU内存温度曲线、谁的健康分最低 |
| 数据质量 | 数据有没有丢失/重复/异常、延迟多少、质量打分 |

### 2.2 用户能看到什么

| 区域 | 显示内容 | 视觉效果 |
|------|---------|---------|
| **顶部KPI卡片** | 今日总车流量 / 平均车速 / 拥堵指数 | 大字显示，带环比箭头（↑绿色=好转，↓红色=恶化） |
| **24小时趋势图** | 0-23点每小时车流曲线 | 双峰曲线（早高峰7-9点+晚高峰17-19点），鼠标悬停显示数值 |
| **拥堵热力图** | 区域×小时 的拥堵等级矩阵 | 绿→黄→橙→红 四色渐变，一眼看出哪个区域哪个时段最堵 |
| **TOP10拥堵道路** | 拥堵等级最高的10条路 | 横向柱状图，最长=最堵，点击道路名可下钻查看详情 |
| **车辆类型分布** | 小型车/中型车/大型车占比 | 环形图，鼠标悬停显示数量和百分比 |

### 2.3 用户可以点什么

| 操作 | 效果 |
|------|------|
| 点击**日期选择器** | 切换查看历史某天的数据（默认今天） |
| 点击**区域下拉框** | 只看某个区（新城区/老城区/高新区/开发区） |
| 点击**道路下拉框** | 只看某条路的数据 |
| 点击**TOP10道路**中的某条 | 下钻到该道路的24小时车速/车流详情 |
| 点击**热力图**某个格子 | 筛选该区域+该小时的所有相关图表 |
| 鼠标**悬停图表** | 弹出详细数值 tooltip |

### 2.4 数据来源

```
Hive ADS 层 (天级离线)
  ├── ads_traffic_operation    → KPI卡片 + 拥堵指数
  ├── ads_top_jam_roads        → TOP10道路排行
  ├── dws_road_hour_flow       → 24小时趋势图 + 车速仪表盘
  ├── dws_area_jam_hour        → 拥堵热力图
  └── dim_area / dim_road      → 中文名称映射
```

**刷新方式**：每5分钟自动刷新（Superset定时刷新 + 浏览器轮询）

### 3.2 用户能看到什么（核心体验）

| 特性 | 效果描述 |
|------|---------|
| **数字跳动** | 实时车流量大数字每秒都在变，像股票行情一样 |
| **颜色变化** | 拥堵5级 → 红色背景 + 闪烁；通畅 → 绿色，一眼区分 |
| **自动刷新** | 页面标题显示"最后刷新时间"，每5秒自动更新所有数据 |
| **告警弹出** | 新告警从右侧滑入，带声音提示（浏览器通知） |
| **持续时间** | 每条拥堵路段显示已持续多久，超过30分钟自动升级告警 |
| **曲线实时绘制** | 车流曲线每秒追加新数据点，形成动态波形 |

### 3.3 用户可以点什么

| 操作 | 效果 |
|------|------|
| 点击**暂停自动刷新** | 冻结当前数据方便截图/分析 |
| 点击**拥堵道路**某一行 | 弹出该道路的24小时车速+车流详情 |
| 点击**告警条目** | 展开告警详情（触发条件/时间/建议动作） |
| 切换**区域筛选** | 只看某个区道路的实时路况 |
| 调整**刷新频率** | 可选 5秒 / 10秒 / 30秒 / 手动 |

### 3.4 数据链路（面试重点）

```
终端传感器 (每30s上报车速/车流)
    │
    ▼
Kafka Topic: traffic_status
    │
    ▼
Flink TrafficCongestionDetection (5分钟窗口 + Watermark 30s)
    │
    ├──▶ Redis (车流/车速/拥堵等级) ──▶ Superset 前端 (5秒刷新)
    │
    └──▶ Kafka Alert Topic ──▶ 钉钉推送 + 前端告警列表
```

**端到端延迟：< 5秒**

### 3.5 为什么能实时？

| 环节 | 延迟 |
|------|------|
| 传感器上报间隔 | 30秒 |
| Flink 窗口计算 | < 1秒 (纯内存) |
| Redis 写入 | < 1毫秒 |
| Superset 查询 Redis | < 100毫秒 |
| 浏览器渲染 | < 100毫秒 |
| **总延迟** | **< 5秒 (从数据产生到前端看到)** |

## 7. 面试话术（直接背）

### 7.1 一句话介绍项目

> "我们整个平台最终通过 Superset 搭建了 4 块可视化大屏，包括城市交通总览、实时路况监控、设备运维监控、数据质量监控。实时指标秒级更新，离线指标按小时/天更新，业务人员可以直接在浏览器里查看路况、设备状态、故障告警和数据质量情况。"

### 7.2 前端亮点话术

> "最核心的是实时路况监控面板，数据从 Kafka → Flink → Redis → 前端，端到端延迟不到 5 秒。页面上车流数字像股票一样每秒跳动，拥堵路段自动变红，设备离线立刻弹窗告警。领导可以在总览大屏上一眼看懂全市交通态势，运维可以在设备看板上第一时间定位故障设备。"

### 7.3 技术栈一句话

> "前端用 Apache Superset 做可视化，后端 Kafka+Flink 做实时计算、Hive 做离线数仓、Redis 做实时缓存、DolphinScheduler 做任务调度，整个链路打通了从数据采集到前端呈现的完整闭环。"

### 7.4 为什么不用自研前端？

> "Superset 是 Apache 顶级项目，支持 SQL 即图表、角色权限、定时刷新、图表联动，我们用它的 Dashboard 功能搭大屏就够了，不需要浪费人力自研前端。业务人员拖拽就能自己创建图表，降低了对开发人员的依赖。"
