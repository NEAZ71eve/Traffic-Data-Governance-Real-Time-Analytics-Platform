# AI辅助模块设计与边界文档

---

## 一、模块定位与边界

### 核心原则
AI模块作为**辅助工具**，提供数据开发效率提升和智能分析能力，**不接入核心数仓/实时链路**，可一键关闭不影响核心业务。

```
┌─────────────────────────────────────────────────────┐
│                    AI辅助层                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │ NL2SQL   │  │ 异常检测  │  │ ETL生成  │          │
│  │ 查询助手 │  │ 智能告警  │  │ 模板工具 │          │
│  └──────────┘  └──────────┘  └──────────┘          │
│        │              │              │                │
│        ▼              ▼              ▼                │
│   只读查询      离线分析结果    开发辅助               │
│   ↓ 不影响      ↓ 不影响       ↓ 不影响               │
│  ┌──────────────────────────────────────┐            │
│  │        核心数仓链路（ODS→ADS）        │            │
│  └──────────────────────────────────────┘            │
└─────────────────────────────────────────────────────┘
```

## 二、NL2SQL 查询辅助

### 技术选型
| 维度 | 方案A（轻量） | 方案B（深度） |
|------|-------------|-------------|
| 模型 | SQLNet / RATSQL | ChatGLM-6B + LoRA |
| 部署 | CPU 推理，轻量级 | GPU (6GB+)，需要微调 |
| 准确率 | 常见查询~80% | 复杂查询~90% |
| 适用 | 实习/演示项目 | 生产环境 |
| **推荐** | **采用方案A（轻量）** | 未来扩展 |

### 评估指标

| 查询类型 | 准确率目标 | 验证方式 |
|---------|----------|---------|
| 单表简单查询 | ≥ 90% | 100条测试用例 |
| 多表JOIN查询 | ≥ 80% | 50条测试用例 |
| 聚合分析查询 | ≥ 75% | 50条测试用例 |
| 复杂子查询 | ≥ 60% | 30条测试用例 |
| **综合准确率** | **≥ 85%** | **230条标准测试集** |

### 错误处理
- 当模型无法识别时，返回 `-- 无法识别"xxx"，请人工编写SQL`
- 置信度 < 0.7 时，同时返回 SQL 和人工编写提示
- 所有生成的 SQL 自动标记 `-- AI Generated: {timestamp}`，便于追溯

## 三、数据异常检测（Isolation Forest）

### 技术选型
| 维度 | 方案A（Isolation Forest） | 方案B（LSTM AutoEncoder） |
|------|-------------------------|-------------------------|
| 模型 | sklearn IsolationForest | PyTorch LSTM |
| 训练数据 | 近30天正常数据（已标注） | 近90天时序数据 |
| 训练时间 | ~5秒（10万条） | ~30分钟 |
| 推理延迟 | <10ms | <50ms |
| 可解释性 | 高（特征重要度） | 低（黑盒） |
| **推荐** | **采用方案A** | 未来升级 |

### 训练数据
- 正常数据：近30天无告警、设备健康度 > 80、拥堵等级 1-2 的数据
- 异常数据：人工标注的告警事件、设备故障、流量突增事件
- 标准：正负样本比例 ~10:1，总样本量 ~50万条

### 评估指标

| 指标 | 目标值 | 说明 |
|------|-------|------|
| 召回率 (Recall) | ≥ 95% | 漏检率 < 5%，宁可多报少漏 |
| 精准率 (Precision) | ≥ 90% | 误报率 < 10% |
| F1-Score | ≥ 92% | 召回与精准的综合平衡 |

### 在线推理流程
```
定时任务（每小时）→ 读取最近1小时DWD数据 → 特征工程 →
IsolationForest.predict() → 异常得分 > 阈值？
  ├── 是 → 推送钉钉告警 + 入库 anomaly_log
  └── 否 → 继续等待下一轮
```

## 四、ETL脚本生成

### 技术选型
| 维度 | 方案A（模板引擎） | 方案B（LLM生成） |
|------|-----------------|-----------------|
| 方式 | Jinja2模板 + 元数据驱动 | ChatGLM-6B + Prompt |
| 准确性 | 100%（模板保证） | ~80%（需人工审核） |
| 灵活性 | 低（仅支持预定义模板） | 高（可生成任意SQL） |
| 维护成本 | 需维护模板库 | 需持续优化Prompt |
| **推荐** | **已实现（ai_etl_generator.py）** | 辅助补充 |

### 当前实现
已基于模板引擎实现三类ETL生成：
1. **ODS DDL生成**：vehicle / traffic / device 三种模板
2. **DWD ETL生成**：数据清洗 + 去重 + 类型转换模板
3. **DWS ETL生成**：聚合 + JOIN模板

### 扩展方向
- 从元数据系统自动读取表结构，无需手动传递 columns 参数
- 基于 Hive 分区信息自动生成分区管理 SQL
- 集成到 DolphinScheduler 中，一键提交生成的 ETL 脚本

## 五、AI 模块与核心链路的隔离

### 安全边界
```
核心链路（不允许AI介入）：
  Kafka → Flink 实时计算 → Redis/Hive
  DolphinScheduler ETL 调度
  
AI辅助区域（允许AI介入）：
  ├── 开发环境：SQL生成辅助（不直接执行）
  ├── 离线分析：历史数据异常检测
  └── 查询加速：分析师自然语言查询转SQL
```

### 异常回退策略
- AI模块故障时，不影响核心 ETL 调度、Flink 实时计算
- NL2SQL异常时，降级为手动编写SQL提示
- 异常检测模型失效时，降级为基于规则阈值（metrics_thresholds.json）
- 一键关闭开关：`config/ai_module_config.json` 中 `enabled: false`
