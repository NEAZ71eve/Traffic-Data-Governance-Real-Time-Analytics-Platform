## 1. Flink · Checkpoint 容灾机制

### 涉及代码

**文件**：`TrafficVehicleCount.java` / `DeviceStatusCEP.java` / `TrafficCongestionDetection.java`  
**位置**：三份文件第 50-70 行，配置完全一致

```java
env.enableCheckpointing(5 * 60 * 1000);                                    // ① 间隔
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE); // ② 语义
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(60 * 1000);        // ③ 最小间隔
env.getCheckpointConfig().setCheckpointTimeout(10 * 60 * 1000);            // ④ 超时
env.getCheckpointConfig().setMaxConcurrentCheckpoints(1);                  // ⑤ 并发数
env.getCheckpointConfig().enableExternalizedCheckpoints(
    CheckpointConfig.ExternalizedCheckpointCleanup.RETAIN_ON_CANCELLATION  // ⑥ 保留策略
);
env.getCheckpointConfig().setCheckpointStorage("hdfs://namenode:8020/..."); // ⑦ 存储
env.setStateBackend(new HashMapStateBackend());                             // ⑧ StateBackend
env.setRestartStrategy(RestartStrategies.fixedDelayRestart(3, Time.seconds(60))); // ⑨ 重启策略
```

### 面试问答

#### Q1：enableCheckpointing(5min) 和 minPauseBetweenCheckpoints(1min) 什么关系？会不会矛盾？

**一句话**：5min 是周期，1min 是保护。不矛盾。

**详细解释**：
- `enableCheckpointing(5min)` 表示每 5 分钟发起一次 Checkpoint
- `minPauseBetweenCheckpoints(1min)` 表示一次 Checkpoint 完成后，至少等 1 分钟才发起下一次
- 作用：如果一次 Checkpoint 跑了 4min，距离"下一次"还有 1min 空闲期，防止 Checkpoint 过密拖慢正常处理
- **对比场景**：没有 minPause 的话，5min 周期一到就立刻发起下一次，可能上次还没处理完又来一次

#### Q2：EXACTLY_ONCE 和 AT_LEAST_ONCE 区别？

| 场景 | EXACTLY_ONCE | AT_LEAST_ONCE |
|------|-------------|---------------|
| 实现 | Barrier 对齐，等待所有算子通道对齐后才快照 | Barrier 不对齐，直接快照 |
| 性能 | 稍差（对齐有等待时间） | 更好 |
| 数据一致性 | 每条数据精确一次 | 可能重复（但绝不丢） |
| 适用场景 | 对账、金额计算 | 推荐统计、监控告警 |

#### Q3：RETAIN_ON_CANCELLATION vs DELETE_ON_CANCELLATION？

| 选项 | 取消时 | 适用场景 |
|------|-------|---------|
| RETAIN | 保留 Checkpoint 数据 | 开发调试停作业改代码再恢复；生产环境保留用于回滚 |
| DELETE | 删除 Checkpoint 数据 | 临时作业、不需要恢复的场景 |

**拓展回答**：我选了 RETAIN。开发中经常需要停掉作业改代码，改完从上次的位置继续消费，不用从头读 Kafka 数据。生产环境一般也 RETAIN，方便运维回滚到某个已知一致的状态。

#### Q4：Checkpoint 和 Savepoint 什么区别？

| | Checkpoint | Savepoint |
|--|-----------|-----------|
| 触发方式 | 自动周期触发 | 手动 trigger |
| 生命周期 | Flink 管理 | 用户管理 |
| 设计目的 | 故障恢复 | 运维操作（升级、修改并行度、调整代码） |
| 版本兼容 | 不保证跨版本 | 设计上跨版本兼容 |
| 存储位置 | 配置的 checkpoint storage | 用户指定的路径 |

#### Q5：HashMapStateBackend 和 EmbeddedRocksDBStateBackend 的区别？你怎么选的？

```java
env.setStateBackend(new HashMapStateBackend());      // 内存
// env.setStateBackend(new EmbeddedRocksDBStateBackend());  // 被注释掉了
```

| | HashMapStateBackend | RocksDBStateBackend |
|--|---------------------|---------------------|
| 存储 | JVM Heap 内存 | 本地磁盘（RocksDB） |
| 状态大小 | 适合小状态（<GB） | 适合大状态（GB~TB） |
| 性能 | 高（内存读写） | 中等（序列化+磁盘IO） |
| GC 压力 | 大（状态大时 GC 频繁） | 小（状态在 RocksDB 中） |
| Checkpoint 方式 | 全量快照 | 增量快照（只传变更） |

**你的回答**："这个项目数据量不大，状态装得下，所以用了 HashMapStateBackend。RocksDB 那行我注释掉了但留着，生产环境状态大时可以切过去。"

#### Q6：作业挂了，恢复过程是怎样的？

1. Flink 检测到 Task 失败
2. JobManager 根据 `restart-strategy` 决定是否重启
3. 如果重启，从最近一次**成功的 Checkpoint** 恢复
4. 所有算子恢复到 Checkpoint 时刻的状态
5. Source 算子恢复到 Checkpoint 时刻记录的 offset
6. 从那个位置重新消费 Kafka 数据

**代码对应**：`fixedDelayRestart(3, 60s)` — 60 秒内最多重启 3 次，之后作业彻底失败需要人工介入。

#### Q7：实际遇到过 Checkpoint 的问题吗？

"遇到过。第一次配的是 `hdfs://namenode:8020/...`，提交作业时报 `UnsupportedFileSystemSchemeException: Could not find file system implementation for scheme 'hdfs'`。原因是 Flink 容器里没有 Hadoop 依赖包，识别不了 hdfs 协议。临时改成 `file:///tmp/flink-checkpoints` 先跑通了。后面要配回 HDFS 的话需要把 Hadoop JAR 加到 Flink 的 lib 目录下。"

**面试官评价**：这是真动手踩过坑的回答，加分。

## 2. Flink · Kafka Consumer 与 Exactly-Once

### 涉及代码

**文件**：`TrafficVehicleCount.java` 第 72-78 行

```java
Properties props = new Properties();
props.setProperty("bootstrap.servers", "traffic-kafka-1:9092");
props.setProperty("group.id", "traffic_vehicle_group");
props.setProperty("enable.auto.commit", "false");
props.setProperty("auto.offset.reset", "earliest");

DataStream<String> stream = env.addSource(
    new FlinkKafkaConsumer<>("traffic_vehicle", new SimpleStringSchema(), props)
);
```

### 面试问答

#### Q1：为什么要关掉 enable.auto.commit？

"Flink 不依赖 Kafka 的自动提交来记录消费进度。它靠 Checkpoint：Checkpoint 时把当前消费的 offset 一起快照到持久化存储。作业挂了重启，从最近一次 Checkpoint 恢复，同时也恢复了当时的 offset，就从那个位置重新消费。如果开着 auto.commit，Kafka 自己提交的 offset 可能和 Flink Checkpoint 里的不一致，恢复时可能出现丢数据或重复消费的情况。"

#### Q2：auto.offset.reset=earliest 什么时候生效？

"只在第一次启动且没有 Checkpoint 状态时生效。后续重启都从 Checkpoint 恢复 offset，这个配置就无效了。"

#### Q3：Flink 怎么保证 Kafka 端到端的 Exactly-Once？

"Source 端靠 Checkpoint 保存 offset，Sink 端分为两种情况：

- **幂等 Sink**（如 Redis HSET）：即使重复写入，结果一致
- **事务 Sink**（如 Kafka Producer）：Flink 两阶段提交协议，Checkpoint 完成时才 commit，否则 rollback"

#### Q4：Kafka 消费 lag 变大怎么排查？

"查三方面：
1. Flink 并行度是否和 Kafka 分区数匹配？不匹配会造成某些 TM 过载
2. Checkpoint 是否太频繁导致处理暂停？
3. 下游 Sink 是否成为瓶颈（如 Redis 写入慢）？"

## 3. Flink · Watermark 与时间语义

### 涉及代码

**文件**：`TrafficVehicleCount.java` 第 97-101 行

```java
.assignTimestampsAndWatermarks(
    WatermarkStrategy.<Tuple3<String, Long, Integer>>forBoundedOutOfOrderness(Duration.ofSeconds(30))
        .withTimestampAssigner((element, recordTimestamp) -> element.f1)
        .withIdleness(Duration.ofSeconds(5))
)
```

### 面试问答

#### Q1：三种时间语义的区别？

| | EventTime | ProcessingTime | IngestionTime |
|--|-----------|----------------|---------------|
| 定义 | 数据产生的时间 | 数据被处理的时间 | 数据进入 Flink 的时间 |
| 确定性 | 不确认（可能乱序到达） | 确定 | 相对确定 |
| 乱序 | 需要 Watermark 处理 | 天然有序 | 有少量乱序 |
| 适用场景 | 关注业务发生时间的场景（车流统计） | 关注处理速度的场景（监控告警） | 折中场景 |

"这里用 EventTime，因为车流统计关心的是车经过卡口的时间，不是 Flink 什么时候处理这条数据。"

#### Q2：Watermark 是什么？你这里的 30s 怎么来的？

"Watermark 是一种衡量 EventTime 进度的机制，本质是一个时间戳，表示"这个时间之前的数据都到齐了"。当 Watermark 超过了窗口的结束时间，就触发窗口计算。"

"30s 是经验值，数据从传感器到 Kafka 再到 Flink，网络延迟一般几秒到半分钟。设得太小（如 5s）大量数据被丢进迟到队列，设得太大（如 300s）窗口要等 5 分钟才出结果，实时性太差。"

**灵魂拷问**："30s 这个数字你是怎么测出来的？"
→ "说实话，我现在是估算的。生产环境应该看真实数据的延迟分布百分位，比如 P99 延迟是 10s 就设 15s。我项目里是模拟数据，所以先用经验值设了 30s。"

#### Q3：withIdleness(5s) 是干什么的？

"空闲检测。数据按 roadId 分组后，如果某条路在某个时段没车经过，这个分区的 Watermark 就永远不推进。Flink 的 Watermark 是所有分区的最小值，一个分区卡住了，所有窗口都不触发。5s 空闲超时表示某个分区 5s 没数据就标记为空闲，不再参与 Watermark 对齐。"

**不设这个会怎样？**
"假如凌晨某条路半小时没车，所有道路的窗口都等半小时才触发。"

#### Q4：迟到数据你怎么处理的？

"代码第 111-113 行注释了侧输出流（Side Output）的代码。现在超时 30s 的数据会被直接丢弃。生产环境应该接到 HDFS，等离线 ETL 补录。"

private static class VehicleCountAggregate implements AggregateFunction<...> {

### 面试问答

#### Q1：Flink 有哪些窗口类型？各适用于什么场景？

| 窗口类型 | 特点 | 你项目里的使用 |
|---------|------|--------------|
| 滚动窗口 Tumbling | 不重叠，固定周期 | 车流统计（5min 一次） |
| 滑动窗口 Sliding | 重叠，有滑动步长 | 如果要做"每分钟输出过去 5min 的车流" |
| 会话窗口 Session | 按不活动间隔划分 | 用户连续访问行为分析 |

#### Q2：AggregateFunction 和 ProcessWindowFunction 区别？

| | AggregateFunction | ProcessWindowFunction |
|--|-----------------|---------------------|
| 计算方式 | 增量聚合 | 全量聚合 |
| 内存占用 | 小（一个累加器） | 大（缓存所有数据） |
| 性能 | 高 | 低 |
| 适用场景 | 简单的聚合（sum、count、avg） | 需要访问窗口内所有数据的复杂计算 |

**一句话**："增量聚合来一条处理一条，全量聚合等窗口关了一次性处理所有。这里用 AggregateFunction 做 count 和 sum 就够了，内存也小。"

#### Q3：滚动窗口和滑动窗口选哪个？如果让你每 1 分钟输出过去 5 分钟的车流，怎么实现？

"用滑动窗口：`.timeWindow(Time.minutes(5), Time.minutes(1))`。代价是计算量是滚动窗口的 5 倍（每 1 分钟触发一次，每次算 5 分钟窗口的数据），但实时性更好。"

#### Q4：merge() 方法什么时候被调用？

"只有使用**会话窗口（Session Window）** 时 merge() 才会被调用。会话窗口的两个窗口合并时，需要把累加器也合并。滚动窗口和滑动窗口不存在合并，merge() 用不到。"

#### Q5：keyBy 之后窗口在不同分区上怎么对齐的？

"按 key 分组后，相同 key 的数据进入同一个分区。窗口计算是基于 EventTime 的，所有分区的窗口用相同的算法对齐（比如 5min 滚动窗口的边界是 xx:00-xx:04, xx:05-xx:09）。Watermark 起的作用就是告诉所有分区什么时候可以触发窗口计算了。"

### 面试问答

#### Q1：CEP 的 next / followedBy / followedByAny 有什么区别？

这是 CEP 面试最最最常考的问题。

| 操作符 | 语义 | 示例 |
|--------|------|------|
| `.next("name")` | 严格连续 | A 后面必须紧跟着 B，中间不能有任何事件 |
| `.followedBy("name")` | 非严格连续 | A 后面跟着 B，中间可以有其他事件 |
| `.followedByAny("name")` | 非确定松散连续 | A 后面跟着 B，中间可以有其他事件，且 B 可以被重复匹配 |

**举例**：数据序列 A, C, B

- `begin("a").next("b")` → **不匹配**（A 和 B 中间有 C）
- `begin("a").followedBy("b")` → **匹配**（A 后面跟着 B，C 被跳过）
- `begin("a").followedByAny("b")` → **匹配**，且如果后续还有 B 也会匹配

**你的规则**：用 `.next()` 的原因是"连续3次离线"，必须依次发生，中间不能有在线状态的记录。逻辑上是对的。

#### Q2：timesOrMore(10).consecutive() 和 times(10) 有什么区别？

- `times(10)`：正好匹配 10 次
- `timesOrMore(10)`：匹配 10 次或以上
- `timesOrMore(10).consecutive()`：匹配 10 次或以上，且必须连续命中

#### Q3：PatternSelectFunction 的 pattern 参数里是什么？

```java
public String select(Map<String, List<DeviceStatus>> pattern) {
    DeviceStatus first = pattern.values().iterator().next().get(0);
    // pattern: { "offline_1"=[DS1], "offline_2"=[DS2], "offline_3"=[DS3] }
}
```

"Map 的 key 是 Pattern 中每个阶段的名字（如 offline_1, offline_2），value 是这个阶段匹配到的事件列表。`first()` 模式只匹配一条记录，所以 List 大小是 1。"

#### Q4：within(Time.seconds(180)) 是基于什么时间的？

"基于 EventTime。结合 assignTimestampsAndWatermarks 设置的 Watermark。Watermark 超过窗口结束时间触发匹配超时判断。"

#### Q5：这个 CEP 代码在集群上跑过吗？

"代码跟着教程写完了，但还没在集群上运行过。目前在 Flink 集群上跑的是车流统计那个作业。这两个 CEP 和拥堵检测的作业代码已写完，正在准备提交。"

## 6. Flink · KeyedState 与 Timer

### 涉及代码

**文件**：`TrafficCongestionDetection.java` 第 138-191 行

```java
private static class FlowAnomalyDetector extends KeyedProcessFunction<...> {
    private ValueState<Double> historicalAvgFlow;  // 历史流量均值
    private ValueState<Integer> sampleCount;        // 样本数

    public void open(Configuration parameters) {
        historicalAvgFlow = getRuntimeContext().getState(
            new ValueStateDescriptor<>("historical-avg-flow", Double.class));
        sampleCount = getRuntimeContext().getState(
            new ValueStateDescriptor<>("sample-count", Integer.class));
    }

    public void processElement(value, ctx, out) {
        // 流量突增检测：当前流量 > 历史均值 * 2
        if (currentFlow > historyAvg * 2 && count >= 5) {
            out.collect("ANOMALY|...");
        }
        // 指数移动平均更新（EMA, alpha=0.2）
        double newAvg = historyAvg * 0.8 + currentFlow * 0.2;
        historicalAvgFlow.update(newAvg);

        // 注册 24 小时后清理 Timer
        ctx.timerService().registerProcessingTimeTimer(ctx.timestamp() + 24*60*60*1000);
    }

    public void onTimer(timestamp, ctx, out) {
        historicalAvgFlow.clear();
        sampleCount.clear();
    }
}
```

#### Q2：ValueStateDescriptor 是干什么的？

#### Q3：KeyedProcessFunction 有哪三个方法？调用顺序是？

#### Q4：ProcessingTimeTimer 和 EventTimeTimer 的区别？

| | ProcessingTimeTimer | EventTimeTimer |
|--|-------------------|----------------|
| 触发条件 | 机器时间到达指定值 | Watermark 到达指定值 |
| Checkpoint | 不会恢复 | 会恢复 |
| 适用场景 | 这里的"24小时后清理" | 比如"等待 30 秒后另一条数据没到" |

**你这里用 ProcessingTimeTimer 对不对？**
"在这个场景里，无论有没有数据来，24 小时后我都希望清理状态，所以用 ProcessingTimeTimer 是对的。如果要等数据（比如"连续 5 分钟没收到心跳才算离线"），应该用 EventTimeTimer。"

#### Q5：EMA（指数移动平均）为什么 alpha 选 0.2？

"0.2 的意思是每次更新时保留 80% 的历史信息，加入 20% 的新信息。alpha 越大对新数据响应越快，0.2 是比较平滑的取值。生产环境可能需要根据流量波动特性调参。"

#### Q6：状态一直增长怎么办？

"我注册了 24 小时的 Timer 清理。生产环境还应该配 State TTL：

```java
StateTtlConfig ttlConfig = StateTtlConfig.newBuilder(Time.days(7))
    .setUpdateType(StateTtlConfig.UpdateType.OnCreateAndWrite)
    .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)

## 8. Hive · 数仓分层与 ETL 清洗

### 涉及代码

**文件**：`sql/dwd/dwd_vehicle_pass_di.sql`（81 行）

```sql
CREATE TABLE IF NOT EXISTS traffic_db.dwd_vehicle_pass_di (
    vehicle_id      STRING,   road_id       STRING,        device_id     STRING,
    pass_time       TIMESTAMP, speed         INT,           direction     STRING,
    plate_number    STRING,   vehicle_type  STRING,        lane          INT,
    hour            INT,      speed_km_per_s DECIMAL(10,4)
)
PARTITIONED BY (dt STRING)
STORED AS ORC TBLPROPERTIES ('orc.compress' = 'SNAPPY');

INSERT OVERWRITE TABLE traffic_db.dwd_vehicle_pass_di PARTITION (dt)
SELECT
    vehicle_id, road_id, device_id,
    CAST(pass_time AS TIMESTAMP)        AS pass_time,
    CAST(speed AS INT)                  AS speed,
    COALESCE(direction, 'UNKNOWN')      AS direction,
    CASE WHEN vehicle_type IN ('小型车','中型车','大型车') THEN vehicle_type ELSE '其他' END AS vehicle_type,
    HOUR(pass_time)                     AS hour,
    ROUND(speed / 3600.0, 4)           AS speed_km_per_s,
    dt
FROM (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY vehicle_id, pass_time ORDER BY pass_time) AS rn
    FROM ods_vehicle_pass_di
    WHERE speed >= 0 AND speed <= 200
) t WHERE rn = 1;
```

### 面试问答

#### Q1：数仓为什么要分层？你这个项目分了哪几层？

"分层的核心目的是**解耦和复用**：

| 层 | 全称 | 职责 | 你的项目 |
|----|------|------|---------|
| ODS | 操作数据存储 | 原样接入，不做清洗 | TEXTFILE 格式保留原始数据 |
| DIM | 维度层 | 统一维度信息 | SCD2 拉链表 |
| DWD | 明细层 | 数据清洗、去重、标准化 | 类型转换 + 值域校验 + 去重 |
| DWS | 汇总层 | 轻度聚合 | 按道路+小时聚合 |
| ADS | 应用层 | 面向业务的指标 | 健康评分、排行榜 |

每一层只做一件事，出了问题可以定位到某一层修。"

#### Q2：DWD 做了哪几种清洗？

"5 种：

1. **类型转换**：pass_time 从 STRING → TIMESTAMP，speed 从 STRING → INT
2. **空值处理**：`COALESCE(direction, 'UNKNOWN')`
3. **范围校验**：`WHERE speed >= 0 AND speed <= 200`，异常数据过滤掉
4. **枚举归一**：`CASE WHEN vehicle_type IN ('小型车','中型车','大型车') THEN '...' ELSE '其他'`
5. **字段派生**：`HOUR(pass_time)` 提取小时，`speed/3600` 计算每秒速度"

#### Q3：为什么 DWD 用 ORC 而 ODS 用 TEXTFILE？

"ODS 用 TEXTFILE 是**保留原始数据**，方便排查问题也能直接查原始内容。DWD 用 ORC 是**列式存储 + 压缩**，查询只读需要的列，IO 少，Snappy 压缩省空间。ODS → DWD 就是"原始格式 → 面向分析的格式"的转换。"

#### Q4：ROW_NUMBER() 去重比 DISTINCT 好在哪？

"DISTINCT 是整行去重，但如果有两个字段完全相同的记录，其中一个的其他字段不同（比如来自不同采集设备），DISTINCT 去不掉。ROW_NUMBER() 可以明确控制保留哪一条——按 pass_time 排序取第一条。"

**追问**："如果同一台车在同一时间有两张照片，车牌不一样，怎么办？"
"那可能是套牌车或者识别错误，应该标记出来交给下游人工处理，而不是直接去重。所以这里 ROW_NUMBER 保留一条是简化处理。"

### 面试问答

#### Q1：什么是动态分区？nonstrict 是什么意思？

"动态分区就是不手动指定写入哪个分区，Hive 根据 SELECT 里分区字段的值自动决定写入哪个 HDFS 目录。`nonstrict` 允许所有分区字段都是动态的（严格模式要求至少有一个静态分区）。"

**举例**：`PARTITION (dt)` 不加具体值，dt 的值来自 SELECT 的最后一列，Hive 按这个值建目录。

#### Q2：动态分区有什么潜在问题？

"三个：小文件爆炸、数据倾斜、`Too many dynamic partitions` 内存溢出。可以通过三个参数控制：

```sql
SET hive.exec.max.dynamic.partitions = 1000;        -- 单次最多创建的分区数
SET hive.exec.max.dynamic.partitions.pernode = 100;  -- 每个节点最多创建的分区数
SET hive.exec.max.created.files = 100000;             -- 一次 MR 最多创建的文件数
```"

## 10. Hive · 两阶段聚合解决数据倾斜

### 涉及代码

**文件**：`sql/dws/dws_road_hour_flow.sql` 第 56-86 行

```sql
-- 阶段1：加随机前缀打散热点key → 局部聚合
GROUP BY CONCAT(CAST(FLOOR(RAND() * 10) AS STRING), '_', road_id), hour, dt

-- 阶段2：去掉前缀 → 再次聚合
SUBSTR(skew_key, 3) AS road_id
GROUP BY SUBSTR(skew_key, 3), hour, dt
```

### 面试问答

#### Q1：数据倾斜什么表现？怎么判断？

"Reduce 阶段 99% 的 task 秒完，一个 task 跑十几分钟不结束。查看 step 日志能看到某条路（如长安街）的数据量是普通路的 100 倍。项目中 benchmark_skew.py 验证过：50 条路 10 万条数据，5 条热点路占 80%，Gini 系数 0.74。"

#### Q2：两阶段聚合的原理是什么？

"**第一阶段**：给 key 加随机前缀（0~9），把"长安街"变成"3_长安街""7_长安街""9_长安街"...，原本一个 Reduce 处理的 5 万条数据，被打散到 10 个 Reduce 各处理 5000 条。

**第二阶段**：去掉前缀还原 key，再做一次全局聚合。因为第一阶段每个 reduce 已经局部聚过了，所以第二阶段只有 10 条中间数据要聚合，没有压力。"

#### Q3：除了加盐还有什么倾斜优化方法？

"Hive 自带 `hive.optimize.skewjoin=true`（第 40 行），自动检测超过阈值的倾斜 key 单独处理。`hive.skewjoin.key=100000`（第 41 行）设置倾斜阈值为 10 万条。"

## 11. Hive · SCD2 拉链表

### 涉及代码

**文件**：`sql/dim/dim_road_zip.sql`

```sql
CREATE TABLE dim_road_zip (
    road_id     STRING,
    road_name   STRING,
    ...
    start_time  STRING COMMENT '拉链生效时间',
    end_time    STRING COMMENT '拉链失效时间，9999-12-31表示当前有效',
    is_current  STRING COMMENT 'Y/N'
)
PARTITIONED BY (dt STRING);

-- 查询当前版本
WHERE is_current = 'Y' AND dt = '${date}';
-- 查询历史版本
WHERE '2026-01-15' BETWEEN start_time AND end_time;
```

### 面试问答

#### Q1：为什么需要 SCD2？全量快照表不行吗？

"假设场景：长安路的限速从 60 改成 80 了。全量快照表每天存一份全量，改前的数据就丢了。需要回溯历史（"2025 年 6 月长安路限速多少"）就查不到。拉链表通过 `start_time / end_time / is_current` 记录每次变更的版本，既能查当前又能溯历史，存储也只存变更记录，节省 90%+ 空间。"

#### Q2：拉链表的增量更新过程是怎样的？

"大致逻辑：

1. **未变更的**：原样保留
2. **变更的**：旧版本闭合（`end_time = 昨天, is_current = 'N'`），新版本插入（`start_time = 今天, end_time = '9999-12-31', is_current = 'Y'`）
3. **新增的**：直接插入当前版本
4. 用 UNION ALL 合并以上结果"

#### Q3：拉链表和全量快照表对存储和查询的影响？

| | 拉链表 | 全量快照 |
|--|-------|---------|
| 存储 | 只存变更 + 当前快照×天数 | 每天存一份全量 |
| 空间 | `变更数×天数 + 当前量` | `全量×天数` |
| 查当前 | `WHERE is_current = 'Y'` | 快照表直接查 |
| 查历史 | `WHERE 'date' BETWEEN start AND end` | 各天快照 UNION ALL |
| 适用范围 | 属性变更不频繁的维度 | 频繁变动的维度或无追踪需求 |

## 14. 综合场景题

#### Q1：数据量变大（比如从 1 千条/天 → 100 万条/天），哪里会先遇到瓶颈？怎么扩展？

"按优先级：

1. **Kafka 分区数**：当前 3 分区不够，加到 8 分区匹配 Flink 并行度
2. **Flink 并行度**：当前 4，可以调大，同时 TaskManager 的 Slot 要够
3. **Hive 小文件问题**：ODS 层 TEXTFILE 会产生大量小文件，如果要入 Hive 需要先合并
4. **StateBackend**：状态超 GB 需要从 HashMap 切成 RocksDB
5. **DWS 数据倾斜**：数据量大了倾斜会更严重，两阶段聚合参数需要重新调"

#### Q2：Flink 作业处理延迟突然从 5s 变成 5min，怎么排查？

"三步排查法：
1. 看 Flink WebUI 的 Watermark 延迟——如果 Watermark 不推进，大概率是某个分区空闲了（`withIdleness` 没配好或某个 Kafka 分区卡了）
2. 看 Checkpoint 耗时和成功率——Checkpoint 频繁失败会导致作业不断重启
3. 看 GC 情况和 RocksDB 读写延迟——状态太大时这是瓶颈"

#### Q3：你的这个项目和别人同类项目比，核心差异是什么？

"说实话，差异不在技术上。大部分大数据项目都类似——Kafka 接数据、Flink 算、Hive 存。我的差异在于：

1. **全部真实部署过**，不是只写了代码
2. **踩过坑**，Hive 连不上 HDFS、Flink 版本冲突、JOIN 条件漏了
3. **数据都真实跑通过**，不是理想状态下的 demo
4. **GitHub 完整开源**，代码、测试、文档全在"
